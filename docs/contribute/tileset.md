# 贴图贡献

:::info[两种角色]
**美术贡献者**负责认领、绘制和填写工作表；**归置维护者**负责运行脚本、审查结果并提交贴图仓库。
:::

CCB 游戏数据定义“应该显示什么”，CCB_UNDEAD_PEOPLE 贴图仓库定义“这些 ID 显示成什么”。贴图贡献因此会同时接触两个仓库，但最终素材通常进入贴图仓库。

## 开始前理解三个概念

- **游戏 ID**：物品、怪物、地形或家具在 JSON 中的稳定标识。
- **sprite**：显示某个 ID 的 PNG 像素图。
- **贴图定义**：把游戏 ID 与 sprite 关联的 JSON。

完整格式、旋转、多格和组合规则以 CCB 仓库 `doc/TILESET.md` 为准。

## 美术贡献流程

1. 从维护者提供的最新追踪工作表中选择状态为缺失的条目。
2. 在群内认领，避免两个人同时绘制同一 ID。
3. 查看游戏 JSON、同类贴图和物品说明，理解尺寸、朝向与用途。
4. 使用硬边像素工具绘制 PNG，不进行插值缩放，不擅自改变规定画布。
5. 把图片嵌入目标行的“提交”列，填写完成状态和贡献者名称。
6. 交回工作表，按检测表反馈修正尺寸、行号或缺失信息。

:::warning[工作表按图片锚点识别行]
图片必须放在目标条目的提交单元格内。图片看起来“盖住了那一行”并不代表锚点正确。
:::

## 维护者工具链

CCB 主仓库提供覆盖率和工作表相关工具；CCB_UNDEAD_PEOPLE 仓库当前包含：

| 工具 | 用途 |
|---|---|
| `tools/gfx_tools/tileset_workbook.py` | 生成或处理追踪工作表 |
| `tools/gfx_tools/ingest_contributions.py` | 校验并归置工作表贡献 |
| `tools/gfx_tools/list_tileset_ids.py` | 列出贴图已有 ID |
| `tools/gfx_tools/compose.py` | 将散图组合为游戏使用的贴图包 |

先查看每个脚本的 `--help`，再执行 dry-run：

```bash
python3 tools/gfx_tools/ingest_contributions.py contribution.xlsx --repo . --dry-run
```

确认检测表中所有目标行合格后再去掉 `--dry-run`。替换已有贴图属于更高风险操作，应由维护者确认并保留原图备份。

## 审查要点

- ID 与工作表类别正确，图片尺寸属于当前贴图包允许范围。
- 透明背景、像素边缘、朝向与同类素材一致。
- 新增贴图定义没有重复 ID；替换模式不会生成重复定义。
- compose 成功，并在游戏中检查正常比例、层级和旋转。
- 作者信息与素材许可满足贴图仓库要求。

## 完成标志

美术贡献以检测表通过为最低标准；最终合入前还必须 compose 并在游戏中查看。遇到尺寸选择、风格或 ID 判断问题，先到 [CCB 不死人贡献群](/community)发参考图与目标 ID。
